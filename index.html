Case Study · Enterprise SaaS

What if Intercom Fin routed on trust state,
not sentiment score?

Fin's current escalation logic detects frustration, repeated loops, and explicit requests for human help — all behavioural proxy signals. This prototype explores what changes when the system first asks why trust is low, before deciding what to do next.

Trust-Sensitive Escalation
Human-in-the-Loop Design
B2B SaaS · Customer Support AI
Intercom Fin · Redesign Concept
The gap
Fin escalates when it detects negative sentiment or complexity. Neither signal tells you why the user doesn't trust the answer — which determines what to do next.
The hypothesis
Brand skepticism and self-ambiguity look identical in sentiment scores but need opposite responses. Collapsing them into one escalation path gets one group wrong every time.
The design decision
Before escalating, the system inserts a trust classification gate — a single question that separates these two states and routes them differently.

Decision rationale
Three design decisions. Each one defended.
Every fork, judgment criterion, decision, and failure condition — documented.
DECISION 01
Trust classification gate before escalation
The fork
Route all low-trust signals to immediate human escalation (Fin's current logic) vs. insert a classification question first.
Criterion
Brand skepticism and self-ambiguity look identical in sentiment scores but need opposite responses. Collapsing them into one path gets one group wrong every time.
Decision
Insert classification gate. Gave up: simpler logic, lower build cost, consistent agent workflow.
Fails when
15 consecutive low-trust cases show <10pp difference in response rate between the two paths → collapse back to single escalation.
DECISION 02
System declines to answer direction-unclear users
The fork
Fin attempts an answer using available knowledge vs. system refuses to answer and asks a clarifying question first.
Criterion
An answer to the wrong question increases confusion, not confidence. In high-cost B2B decisions, a misaligned answer is worse than no answer.
Decision
Decline to answer. Gave up: faster time-to-first-response, higher resolution rate metric in the short term.
Fails when
Users flagged as direction-unclear who receive a clarifying question have a lower satisfaction score than those who received a direct answer → revert.
DECISION 03
Hidden stakeholder detection triggers AE brief
The fork
Continue conversation as normal vs. detect "need to check with [title]" language and flag the deal with a stakeholder brief for the AE.
Criterion
Hidden stakeholders are the single most common reason B2B deals stall after a strong AI-assisted discovery. If the system can detect the signal, not flagging it is a product failure.
Decision
Flag and brief AE. Gave up: uninterrupted Fin conversation flow; risk of false positives creating noise for AEs.
Fails when
AE brief false positive rate >30% (AEs report irrelevant flags) → tighten detection threshold before re-enabling.
CORE BELIEF
"AI choosing not to answer is a trust signal — not a system failure."
Fin's bet
AI coverage maximisation — human is the fallback when AI can't resolve.
This bet
Trust state determines escalation — not problem complexity. An AI answer to a low-trust user in a high-stakes B2B decision makes things worse, not better.
Validated by
Decision Confidence System beta — 80% lead-to-consultation conversion using trust-state routing (n=5, invited sample).
